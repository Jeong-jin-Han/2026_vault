## Abstract
This paper argues that Transformers lack a native mechanism for knowledge lookup, forcing them to approximate retrieval through expensive computation.
By introducing conditional memory (Engram), the authors separate look up from computation, revealing a new sparsity trade-off between neural computation and static memory
## Introduction

This paper argues that while sparsity is a long-standing principle in intelligent systems, modern LLMs realize it almost exclusively through conditional computation (MoE).
However, language modeling consists of two fundamental different sub tasks:
dynamic compositional reasoning and static knowledge retrieval.
Since Transformers lack a native lookup primitive,
current models are forced to simulate retrieval through expensive computation,
wasting early-layer capacity on trivial operations.
To address this mismatch, the authors propose conditional memory as a complementary sparsity axis, instantiated by Engram.
By formulating the Sparsity Allocation problem, they show that properly balancing computation and memory leads to significant gains in reasoning, long-context understanding, and system efficiency.



#paper/contents
# Prior Work
Sparsity has long e been a fundamental design principle in intelligent systems, ranging from biological neural circuits to modern Large Language Models (LLMs).
In contemporary LLMs, sparsity is primarily realized through Mixture-of-Experts (MoE), which scales model capacity via conditional computation, activating only a subset of parameters per input. This approach enables significant model scaling without proportional increases in computation and has therefore become the de facto standard for frontier models.
At the same time, classical N-gram models demonstrated that many local and repetitive linguistic patterns can be efficiently handled through simple lookup mechanisms rather than deep computation. However, modern Transformer architectures lack a native mechanism for such knowledge lookup, relying almost exclusively on computation-based approaches.

# Motivation
Language modeling inherently consists of two qualitatively different sub-tasks:
1. Computational reasoning, which requires deep, dynamic computation.
2. Knowledge retrieval, which often involves local, static, and highly stereotyped patterns
Despite this duality, Transformers treat both tasks uniformly through computation. As a result, even simple retrieval tasks - such as resolving multi-token entities - consume multiple early layers of attention and feed-forward computation. This effectively forces models to simulate retrieval through computation, leading to inefficient use of model capacity and wasting depth that could otherwise be allocated to higher level reasoning.

# Core idea
To address this mismatch, the paper proposes introducing conditional memory performs sparse lookup operations to retrieve static knowledge directly.
As a concrete instantiation of this idea, the authors introduce Engram, a modernized N-gram-based memory module. Engram enables constant-time O(1) lookups using local context as an index into a large embedding table, augmented with techniques such as tokenizer compression, multi-head hashing, contextualized gating, and multi-branch integration. This design treats lookup as a firs-class modeling primitive rather than an emergent behavior learned through computation.

# Results
The authors formalize the Sparsity allocation problem, which asks how a fixed parameter budget should be distributed between MoE experts (computation) and Engram memory (lookup). Their experiments reveal a U-shaped scaling law, showing that neither computation-only nor memory-only extremes are optimal.
Guided by this principle, they scale Engram to a 27B-parameter model and demonstrate superior performance compared to iso-parameter and iso-FLOPs MoE baselines. Performance gains are observed not only in knowledge-intensive benchmarks but also in general reasoning, code, and math tasks, indicating that freeing early layers from static reconstruction benefits higher-level reasoning.

# Author's Limitation
The effectiveness of Engram relies on the assumption that a substantial portion of linguistic patterns can be represented as static, lookup-friendly knowledge. While the results suggest strong empirical benefits, the approach may be less effective for tasks dominated by highly dynamic or novel patterns. 
Additionally, although the system-level design enables efficient host-memory offloading with minimal overhead, the approach introduces new architectural complexity and depends on careful engineering to fully realize its efficiency gains in practice.

## 1. HW/RTL 관점에서 가능한 큰 ㄱ라래 3개
### A) 저장 구조 자체를 바꿔서 중복 제거 (Trie)

#### A1) LOUDS / Succinct Trie (비트벡터 기반)
- 트리를 포인터로 들고 다니며 HW에서 비사고 랜덤 접근이 많아짐

- 장점: 포이터 제거, 메모리 효율, 점근 패턴 규칙적 -> 프리 패치 유리
- 단점: rank/selecte 회로/microcode 필요, 업데이트 어려움
#### A2) Double-Array Trie (DAT)
- base[], check[] 두 배열로 트리 표현
- 자식 탐색이 O(1)에 가까운 배열 접근으로 바뀜
- 장점: 매우 HW friendly (연속 배열 접근), 캐시 효율 좋음
- 단점: build가 까다롭고 재배치 비용 (하지만 모델 배포 후 고정이면 OK)

#### A3) Minimal DFA / DAWG (중복 suffix까지 압축)
- 접두뿐 아니라 suffix / 부분 문자열도 공유
- 장점: 중복 제거 극대화
- 단점: 구축 복잡, 업데이트 거의 불가 -> 역시 정적 테이블에 정합

## 3) LLM-Assist CSL 설계 아이디어 5가지
#### A) Deterministic lookup 기반 "Signature Prefetch"
Engram은 key가 context에서 결정되니, 다음 토큰에서 접근할 주소 후보가 예측 가능함 (shift되는 engram)

아이디어
- gpu/host가 signature를 만들면
- CXL 축 (host-side runtime)이
	- 다음 step의 lookup후보 주소를 미리 prefetch
	- CXL 링크에서 읽기를 앞당겨 latency overlap
도메인 특화 포인트
- 일반 prefetcher (스트라이드/델타)보다
- 토큰 시퀸스 기반 signature prefetcher 가 훨씬 잘 맞습니다.


#### B) CXL 메모리 쪽에 Lookup Cache (SRAM like tier) 붙이기
CXL type -3 메모리 디바이스가 단순 DRAM만 달고 끝나면, 랜덤 lookup latency가 부담될 수 있음
아이디어
- CXL 메모리 디바이스 내부에 
	- 작은 tag + data 캐시 (예: recent signature -> embedding pointer/result)
	- 또는 bloom filter + hot table 캐시를 붙이면 hit율이 높을 때 성능이 크게 된다.
왜 LLM에 잘 맞나?
- 언어는 반복 패턴이 많아서
- 같은 n-gram signature 가 자주 재방문된다.


#### C) Embedding Table 전용 Addressing / Layout
도메인 특화의 핷미은 데이터 레이아웃을 지배하는 것
아이디어
- signature -> ID 매핑을 통해 테이블을
	- 페이지/라인 단위로 연속적으로 배치
- 자주 함께 조회되는 엔트리를 근접 배치
- hugepage (2MB/1GB) 활용 + NUMA pinning
- GPU side cahce line 효율을 고려한 embedding packing

이건 CXL 프로토콜 자체를 바꾸지 않아도,
CXL 메모리를 쓰기 쉬운 형태로 데이터 배치만 잘해도 효과가 큼


#### D) Host memory offload를 정식 캐시 계층으로 만드릭
논문도 "host memory prefetch" + "negligible overhaed"같은 이야기를 한다.

중요 포인트
- GPU가 CXL 영역을 직접 읽을 때는
	- 동시 outstanding read를 많이 걸어 latency 숨ㄱ기
- evicition 정책 (무엇을 HBM에 남길지) 매우 중요

#### E) CXL 위에 올라가는 LLM Runtime ABI 만들기
진짜 도메인 특화는 프로토콜보다 소프투웨어 인터페이스(ABI)가 될 때가 많음

예시 api
- prefetch (signature_batch)
- lookup(signature_batch) -> embedding vector batch 반환
- pin_hotset(hot_ids) -> 특정 범위는 디바이스 캐시에 고정

이렇게 런타임이 내가 LLM이다 라고 말해주면,
CXL 디바이스/호스트는 그에 맞춰 prefetch/캐싱을 최적화할 수 있습니다.

## 4) 현실적인 난점도 같인 짚어야 함
도메인 특화 CXL로 갈 때 항상 나오는 3가지 함정

### (1) 레이턴시
CXL은 용량 확장에는 좋지만, HBM보다 늘빈다.
따라서 prefetch + batching + outstanding으로 레이턴시를 숨기는 설꼐가 거의 필수입니다.

### (2) 랜덤 접근이 너무 많으면 링크가 병목
lookup이 너무 난사되면 CXL 링크가 막힙니다.
그래서 signature를 잘 압축하고 (tokenizer compression) 캐시 hit을 높이는 설계가 중함

CXL.cache/coherence가지 가면 복잡도 커짐
Engram 테치블이 read-mostly 라면, coherence 요구를 최소화하는 방향 (읽기 전용 맵핑, 버전 관리)
